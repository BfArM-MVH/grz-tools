## grz-tools release (2026-05-27)

This release updates multiple packages in the `grz-tools` monorepo. It contains the version bumps and changes summarized below.

> ⚠️ Please update your installations and ensure your LEs upgrade their `grz-cli` version too.

### Versions
- `grzctl` `2.1.1` (from `1.4.0`)
- `grz-cli` `1.7.0` (from `1.5.1`)
- `grz-pydantic-models` `2.7.1` (from `2.5.0`)
- `grz-db` `2.1.1` (from `1.2.0`)
- `grz-common` `2.1.0` (from `1.6.1`)
- `grz-check` `0.3.1` (from `0.2.1`)
- GRZ QC workflow `2.0.0`

Since some of these changes affect the submission workflow, the submission SOP will be updated accordingly as soon as possible.

## ⚠️ Breaking / behavior changes

### grzctl: Automatic DB state updates on command execution (#501)
`grzctl` commands now update the submission state in the DB automatically.

- Default: `--update-db`
- Disable: `--no-update-db`

Behavior:
- Before running the main work, the command writes an in-progress state such as `DOWNLOADING`, `VALIDATING`, `ENCRYPTING`, or `REPORTING`.
- On success, it writes the corresponding finished state.
- On failure, it writes `ERROR` and stores the error message in the state log data.

Affected commands include:
- `grzctl download`
- `grzctl decrypt`
- `grzctl validate`
- `grzctl encrypt`
- `grzctl archive`
- `grzctl clean`
- `grzctl pruefbericht submit`

Additional note:
- The DB state enum now includes `REPORTING` for the in-progress Prüfbericht submission phase.

### grzctl: `pruefbericht submit` requires `--submission-id`
As part of the DB-state-update workflow, `grzctl pruefbericht submit` now requires `--submission-id` so the submission can be updated consistently in the DB.

### Additional behavior-impacting changes

- `grz-cli` `upload`/`submit` now enforce version policies via S3 `version.json`. (#477)
- Submission DB schema/metadata expectations changed: `selected_for_qc` plus QC workflow version and `grzctl` tooling/runtime version logging are now required/recorded. (#531, #561)
- `grzctl db submission populate-qc` now requires `--qc-workflow-version` (or `GRZCTL_QC_WORKFLOW_VERSION`) to be provided so the QC workflow version can be recorded in the DB. (#561)
- `grzctl backfill` can re-read metadata from archives, affecting archive/DB reconciliation workflows. (#559)
- GRZ QC workflow `2.0.0`: `report.csv` now shows deduplicated metrics by default.
- `grz-pydantic-models`/`grz-cli`/`grz-common`: metadata validation adapted to BfArM requirements effective 01.06.2026. Warnings switch to errors on a time-based trigger. (#570)
- Submission failure tracking: `grz-common`/`grz-db` now record a structured failure reason on the submission state when a worker fails; a new migration adds the `failure_reason` column. (#544)
- Transitive workspace dependencies were stripped from `grz-cli`, `grzctl`, and `grz-common` `pyproject.toml`; newly-added direct dependencies are now version-pinned. Environments that relied on previously-transitive packages may need to install them explicitly. (#573, #577)
- `grz-common` no longer depends on `pynacl`; X25519 key generation uses `cryptography` instead. (#572)
- Dropped the Python dependency `pysam` from `grz-cli`, `grzctl`, and `grz-common` (installs are lighter; environments that relied on it being installed transitively must install it explicitly). (#565)

## Migration instructions

### Database (required)
This release adds `selected_for_qc`, QC workflow version logging, `grzctl` runtime version logging, the new `REPORTING` state, the `qc_queue` table, and additional `submissions` fields (`submission_size`, `submission_metadata`). The DB must be upgraded to `head` before using the release.

1. Ensure your database settings reside in `~/config/grz-cli/config.yaml`.
2. Back up your database and run the upgrade: `grzctl db upgrade`.
3. Run the database backfill (this can take a while): `grzctl db backfill --config-file $ARCHIVE_CONFIG`. Run this once for the non-consented archive bucket and once for the consented archive bucket.

### Automation / deployments
- Update any automation that calls `grzctl pruefbericht submit` to pass `--submission-id`.
- If you enforce LE uploads via S3, deploy the `version.json` policy for `grz-cli`. Example policy enforcing `grz-cli` ≥ `1.7.0` starting `2026-06-01`:

  ```json
  {
    "schema_version": 1,
    "grzcli_version": [
      {
        "enforced_from": "2026-06-01T00:00:00+00:00",
        "minimal_version": "1.7.0",
        "recommended_version": "1.7.0",
        "max_version": null
      }
    ]
  }
  ```
- Deploy the matching `grz-check` Python bindings where Python-based checksum validation is used.

For more information, see: https://github.com/BfArM-MVH/grz-tools/releases
